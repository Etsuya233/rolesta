# 项目文档维护

项目文档按用途归档，目录名称保持通用，不与编写工具或工作流绑定。

| 路径               | 职责                                         |
| ------------------ | -------------------------------------------- |
| `README.md`        | 产品定位、当前范围和面向所有读者的项目入口   |
| `AGENTS.md`        | 开发任务入口与必读文档索引                   |
| `docs/development` | 当前有效的工程结构、边界和日常开发约定       |
| `docs/adr`         | 难以逆转且存在真实取舍的长期架构决策         |
| `docs/specs`       | 功能或架构的阶段性设计记录                   |
| `docs/plans`       | 对应设计的阶段性实施计划                     |
| `docs/handoffs`    | 尚需继续处理的工作状态、验证结果和交接上下文 |

设计文档存放在 `docs/specs`，实施计划存放在 `docs/plans`。文档目录按用途命名，不创建以工具或工作流命名的目录。

## 维护原则

当前行为、工程约定与目录职责优先记录在 `README.md`、`AGENTS.md` 和 `docs/development`。spec、plan 与 handoff 保留设计和实施过程，不覆盖当前代码与开发文档形成的事实。

代码与开发文档不一致时，先确认代码是否偏离既定约束，或架构是否已经合理演进。前者修正代码，后者同步更新对应开发文档；涉及难以逆转的取舍时补充 ADR。

新增或移动文档时检查仓库内的 Markdown 相对链接。spec 与 plan 文件沿用 `YYYY-MM-DD-<topic>.md` 命名，相关文档使用相对路径互相引用。完成阶段性工作后保留设计与计划作为历史记录，内容中应避免把尚未实现的能力写成当前事实。
