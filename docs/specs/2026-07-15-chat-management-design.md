# Chat 管理功能设计

## 目标

本阶段为 Chat 模块建立可用的会话管理能力，覆盖创建、分页列表、筛选、查看、编辑和删除。会话可以关联对话角色、Persona、预设和模型连接。本阶段不包含消息、消息输入、LLM 调用、上下文组装或世界书关联。

## 领域模型

聊天会话归创建它的用户所有，且始终仅对所有者可见。会话包含以下字段：

- `id`
- `ownerUserId`
- `title`
- `chatCharacterId`
- `personaCharacterId`
- `presetId`
- `modelProviderId`
- `createdAtMs`
- `updatedAtMs`

创建会话时必须选择一个当前用户可见的对话角色。角色可以来自用户自己拥有的角色卡，也可以来自其他用户的公开角色卡。创建后允许替换对话角色，用户不能主动清空该关联。未来接入消息后，更换对话角色只影响后续回复；已有消息继续保留，且既有消息的作者归属不会随当前对话角色变化。

Persona、预设和模型连接均为可选关联。Persona 可以来自会话所有者自己的角色卡，也可以来自其他用户的公开角色卡；预设同样允许使用本人或其他用户的公开资产。模型连接必须归会话所有者所有。用户可以更换或清空这三个关联。未来接入消息后，当前 Persona 只影响后续用户消息和上下文组装；已有用户消息保留发送时的 Persona 归属，即使当前 Persona 已更换、清空或删除。

已保存的标题始终非空且无需唯一。创建请求必须包含标题；更新请求可以省略标题，提交标题时必须保持非空。共享 `chatTitleSchema` 去除标题首尾空白，再校验非空和最长 512 个字符，标题内部空白原样保留；服务端不根据角色名称生成或补全标题。角色改名或更换不会在服务端自动修改标题。角色卡、预设等现有资产的空名称规则后续单独统一，本阶段不扩大对应模块的修改范围。

`updatedAtMs` 表示会话最近活动时间。创建会话、成功处理非空更新请求，以及未来的消息变化都会更新该时间；服务端不识别同值更新，因此合法的同值请求也会更新时间。资产删除、转为私有或实时关联资产内容变化不会更新该时间，避免旧会话因外部资产变化移动到列表顶部。列表中的“最近更新时间”和默认排序均使用该字段。

## 资产默认值

创建表单读取当前用户的默认 Persona、预设和模型连接，并将它们作为一次性预填值。表单提交后，会话保存自己的独立关联；用户以后修改默认值不会影响已有会话。

现有聊天偏好仍只允许把本人角色卡设为默认 Persona。该限制只约束默认值来源；创建或编辑 Chat 时，用户可以将预填值替换为其他用户当前公开的角色卡。

默认资产读取属于非阻塞的预填步骤。读取失败时仍打开创建表单，Persona、预设和模型连接保持未选择，并在表单内显示默认值加载失败状态与重试命令；对话角色仍由用户选择。重试期间保留用户已经填写的标题和资产选择，成功后只补充用户尚未手动操作的字段。

创建表单把用户默认模型连接视为系统预填值。在用户尚未手动操作模型字段时，默认或随后选择的自有预设如果关联了模型连接，表单使用该关联覆盖系统预填值；用户手动选择或清空模型连接后，该字段进入锁定状态，后续切换预设不再自动覆盖。公开预设不暴露原作者的模型连接，选择公开预设时继续保留当前表单中的模型连接。

编辑表单把 Chat 已保存的非空模型连接视为已锁定，不因切换预设自动覆盖；已保存连接为空时，选择关联了模型连接的自有预设可以自动带入，直到用户手动操作模型字段。创建与编辑表单只要当前选择的是关联了模型连接的自有预设，就在模型候选列表中将该连接置顶，供用户识别和选择；未关联连接的自有预设、公开预设和清空预设均不改变当前模型连接。

服务端接收表单最终选中的资产 ID，并重新验证所有访问权限。默认值优先级属于创建表单行为，不进入 Chat 领域模型。

## 生命周期

聊天会话使用硬删除，并在删除前要求用户确认。本阶段不提供归档、置顶、批量操作、软删除、公开或分享能力。

资产删除时，会话继续存在，对应关联被清空：

- 删除角色卡会清空它作为对话角色或 Persona 的关联。
- 删除预设会清空预设关联。
- 删除模型连接会清空模型连接关联。

其他用户的公开角色卡转为私有后，Chat 模块清除当前用户会话中该角色卡对应的对话角色和 Persona 关联；公开预设转为私有后清除对应预设关联。资产所有者自己的会话关联继续保留。对话角色关联被系统清除后，会话允许暂时缺少对话角色，用户可以通过编辑会话重新选择角色。

缺少对话角色时，用户仍可查看已有消息、编辑标题及其他配置、重新选择角色或删除会话。发送消息、重新生成和继续生成等需要系统角色参与的操作必须禁用，直到用户重新选择对话角色。

会话只保存资产 ID，不保存角色名称、头像、预设名称或模型连接名称的快照。Chat 的当前配置实时引用资产：角色卡、Persona、预设或模型连接的后续修改会影响会话下一次使用这些资产时的行为，其他用户对公开资产的修改也遵循相同规则。接口在读取时返回当前仍有效的资产摘要。

实时关联不改写已有消息及其作者归属。未来生成模块必须单独记录每次生成实际使用的上下文快照，使用户能够解释历史结果；该快照不属于本阶段的 Chat 数据模型。

## 数据库

新增 `chats` 表。四个资产关联均使用可空外键，删除行为为 `ON DELETE SET NULL`。`owner_user_id` 使用 `ON DELETE CASCADE`。

建议索引覆盖：

- `owner_user_id, updated_at_ms, id`
- `owner_user_id, created_at_ms, id`
- `owner_user_id, title, id`
- `owner_user_id, chat_character_id`

列表排序始终追加 `id` 作为稳定次序。标题关键词使用当前 SQLite 列表查询约定实现。

## 后端模块

`apps/api/src/chats` 按现有业务模块结构补充：

- `domain`：Chat 对象、创建与编辑规则。
- `application`：创建、列表、详情、更新、删除用例，以及资产可见性事件监听器。
- `ports`：`ChatStore` 与 Chat 所需的资产访问能力。
- `persistence`：Kysely Store、行映射和资产访问实现。
- `http`：Controller、请求 DTO、响应 DTO 和错误映射。
- `testing`：用例测试需要的明确 fake 或 builder。

Chat 应用层通过窄 port 验证资产访问权，不导入其他模块的 persistence。模块装配层绑定具体实现。创建和更新 Chat 必须在 Unit of Work 中获取并验证所有非空资产关联；PostgreSQL 与 MySQL 对匹配的资产行加锁，SQLite 在同一事务中完成验证与写入。任何成功提交的非空关联在提交时都必须对会话所有者有效。

角色卡和预设模块在资产从公开转为私有时发布过去式领域事件。涉及可见性变化的资产更新与事件发布必须位于同一 Unit of Work；Chat 监听器在当前事务中清除其他用户会话的关联。资产删除、可见性变化与 Chat 关联写入通过事务和资产行锁串行化，监听器失败会使资产更新事务回滚，保持可见性与引用一致。

## HTTP API

新增以下端点：

- `POST /chats`
- `GET /chats`
- `GET /chats/:id`
- `PATCH /chats/:id`
- `DELETE /chats/:id`

创建请求包含必填的 `title` 和 `chatCharacterId`，以及可选的 `personaCharacterId`、`presetId` 和 `modelProviderId`。更新请求支持标题与四项资产关联；提交标题时必须通过同一个 `chatTitleSchema`，`chatCharacterId` 只接受新的角色 ID，另外三个关联接受 `null` 以清空。更新请求必须至少包含一个可更新字段，空对象由共享 Zod schema 拒绝；服务端不额外比较提交值与当前值是否相同，合法的同值请求仍按普通更新处理。

列表接口支持：

- 标题关键词。
- 单一角色筛选器对应的“全部”“缺少角色”“指定角色”三种状态。
- `createdAt`、`updatedAt`、`title` 排序。
- 升序或降序。
- `pageIndex` 和 `pageSize`。

列表默认按 `updatedAt` 倒序，每页 20 条。Web 允许选择 10、20、50 或 100 条。

列表响应保持最小，只包含 `id`、`title`、`updatedAtMs` 和 `chatCharacter`。`chatCharacter` 为 `{ id, name, avatar } | null`；关联被系统清除时返回 `null`。

详情响应包含 `id`、`title`、`chatCharacterId`、`personaCharacterId`、`presetId`、`modelProviderId`、`createdAtMs` 和 `updatedAtMs`，并提供以下最小展示摘要：

- `chatCharacter`：`{ id, name, avatar } | null`
- `persona`：`{ id, name, avatar } | null`
- `preset`：`{ id, name } | null`
- `modelProvider`：`{ id, name, providerKind, defaultModelName } | null`

Chat 响应不复用现有资产管理接口的宽摘要。资产候选、所有者和可见性由对应资产搜索接口提供；Chat 响应不包含角色卡正文、预设正文、Base URL、凭据、统计或其他私有资产内容。

查询不存在或不属于当前用户的 Chat 时返回相同的 404。创建或更新引用无权访问、已经删除或已经失效的资产时，返回稳定的 Chat application error，由 HTTP 层映射为对应错误响应。

资产存在性、所有权和可见性由应用层按对话角色、Persona、预设、模型连接的固定顺序逐项检查，每次返回当前遇到的单个字段错误。缺失、删除、失效和越权使用相同的不可用语义，错误只携带字段名，不回显资产 ID；前端保留其余表单内容，用户修正后可以再次提交。

本阶段不提供 Chat 编辑的乐观锁、版本检查、字段级并发合并或冲突提示。单次 `PATCH` 中省略的字段表示不修改；多个并发更新发生覆盖时，以数据库实际持久化顺序产生的结果为准，相关冲突由用户自行处理。

### 共享请求校验试点

Chat 作为前后端共享 Zod 校验的首个试点模块。`@rolesta/shared` 提供 Chat 创建、编辑和列表查询的 Zod 4 schema，统一定义字符串规范化、长度、必填关系、可空字段、枚举及分页范围等纯字段规则；Web 表单直接使用 schema，API 通过 `nestjs-zod` 从同一 schema 派生请求 DTO、运行时校验和 OpenAPI 描述。Chat 请求不再重复声明 `class-validator` 规则。

现有全局 `ValidationPipe` 属于前期占位实现，本阶段允许替换为可复用的项目级请求校验入口。该入口根据 DTO 的明确元数据选择 Zod 或旧模块的 `class-validator` 校验，禁止同一请求依次执行两套字段校验；Chat 请求只使用共享 Zod schema，现有模块在后续迁移前继续使用原 DTO 规则。实现需要集中处理转换、未知字段拒绝和错误映射，Controller 不手动调用 schema，也不分别拼装校验响应。

共享 schema 一次报告全部纯字段问题。API 将问题映射为稳定的 `{ field, rule }` 列表，字段使用请求字段路径，规则使用项目定义的机器码；响应不包含原始输入值或 Zod 的英文错误文本。Web 使用相同 schema 进行提交前校验，并根据 `field` 与 `rule` 映射本地化文案。字段规则全部通过后，资产权限校验仍按固定顺序逐项执行并每次返回一个字段错误。

OpenAPI 生成流程接入 `nestjs-zod` 要求的文档清理，并通过生成文件断言确认 Chat 的必填、可空、长度和枚举约束没有丢失。资产权限、数据库状态和并发有效性不进入共享 schema，继续由 Chat 应用层判断。

接口完成后重新生成 `apps/api/openapi.json` 与 `apps/web/src/lib/api/generated/schema.ts`，Web 只使用生成契约。

## Web 交互

现有左侧 Chat Context 面板改为两个 Tab：

- “聊天列表”：创建、搜索、筛选、排序、分页和会话选择。
- “当前聊天”：查看当前会话完整配置并执行编辑、删除。

Workbench 只使用 `/app` 路由。活动 Chat ID 由 Workbench 页面生命周期内的内存状态持有，不写入路径、查询参数或本地持久化；刷新页面后清除活动 Chat 并默认打开“聊天列表”。用户选择会话或创建成功后切到“当前聊天”，也可以随时切回列表。窄屏选择会话后关闭左侧抽屉，使中央区域重新可见。

当前占位的 `/app/chats/:chatId` 路由在本阶段移除。角色卡、预设、世界书和模型连接的独立测试路由不属于本阶段的清理范围。未来消息面板需要深链接时，再结合 Center 导航模型统一设计 Chat URL。

聊天列表适配左栏 224–448px 的宽度范围。每条记录展示标题、对话角色头像与名称、最近更新时间，以及缺少角色状态。当前会话高亮，条目菜单提供编辑和删除。Persona、预设和模型连接在“当前聊天”Tab 展示。

筛选区包含标题搜索、单一对话角色筛选器和排序菜单。筛选或排序改变时回到第一页。分页复用 `features/assets/components/page-controls.tsx`，保持其他资产管理器的页码和每页数量交互。

创建与编辑共用一个对话框。桌面端居中展示，移动端占用大部分可用空间。资产选择器支持搜索；对话角色、Persona 和预设搜索本人及公开资产，模型连接只搜索本人资产。创建表单在用户首次手动编辑标题前将标题视为自动预填：选择或切换对话角色时，角色名称会实际写入标题文本框；角色名称为空时标题也保持为空并显示字段错误。用户首次手动编辑标题后，切换角色不再覆盖标题，即使用户将标题清空；空标题不能提交。表单向 API 提交文本框中的最终标题，服务端不执行标题补全。

创建成功后关闭对话框、刷新列表、将新 Chat 设为活动 Chat 并切到“当前聊天”。编辑成功后关闭对话框并刷新受影响的列表与详情，不改变活动 Chat，也不主动切换 Tab；从列表编辑非当前 Chat 时，用户继续停留在列表，原活动 Chat 保持不变。

编辑表单使用规范化后的标题和四项资产 ID 与已加载详情计算脏状态。没有实际变化时禁用保存命令，任一字段变化后才允许提交；该判断属于 Web 交互，不构成服务端并发或幂等保证。

“当前聊天”Tab 展示标题、ID、四项资产摘要、创建时间和更新时间，并提供编辑与删除命令。没有活动会话时显示“未选择聊天”。

活动 Chat 属于工作区上下文，Center 当前打开的面板属于独立布局状态。选择、切换或清除活动 Chat 不自动切换 Center，也不关闭已经打开的测试页或其他 Center 面板；本阶段保留现有工作区首页和 Center panel registry，不增加会话名称与 ID 占位页。未来消息功能通过独立 Center 面板接入。

## 错误状态

列表加载失败时，左栏显示错误状态与重试命令。活动会话详情发生网络错误或返回 404 以外的失败响应时，Web 保留活动 Chat 和“当前聊天”Tab，在详情区域显示错误状态与重试命令。只有详情明确返回 404 时，Web 才清除内存中的活动 Chat 并切回聊天列表。删除当前会话后执行相同的状态重置，无需导航。

创建或编辑期间发生资产失效或访问权变化时，对话框保留用户已经填写的内容，并在对应字段显示错误。提交期间禁用重复提交。

- 列表查询、分页与缓存失效。
- 左栏 Tab 自动切换。
