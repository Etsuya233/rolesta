# Rolesta Internationalization Design

## Context

Rolesta needs project-wide internationalization for the web app and API error responses. The first supported locales are English, Simplified Chinese, Traditional Chinese, and Japanese.

The current codebase is a TypeScript monorepo:

- `apps/web`: React and Vite web app.
- `apps/api`: NestJS API.
- `packages/shared`: shared API envelope and error code types.

The existing API already returns a unified envelope with `code`, `msg`, and `data`. This design keeps that shape and gives `code` a stable cross-end role: API logic identifier and translation key identifier.

## Goals

- Support `en-US`, `zh-CN`, `zh-TW`, and `ja-JP`.
- Use the browser language on first visit.
- Persist a user-selected language in `localStorage`.
- Send the active language with every API request.
- Return API error `msg` in the requested language.
- Let the web app render API errors from `code` and `data`, so visible errors update when the user changes language.
- Require all future user-visible UI text, form validation text, and backend error text to use i18n resources.

## Locale Model

`packages/shared` will define the locale contract:

```ts
export const SUPPORTED_LOCALES = ['en-US', 'zh-CN', 'zh-TW', 'ja-JP'] as const;

export type SupportedLocale = (typeof SUPPORTED_LOCALES)[number];
```

Shared locale normalization will map common browser and request values:

- `en`, `en-US`, and other English tags map to `en-US`.
- `zh-CN`, `zh-Hans`, and mainland Simplified Chinese tags map to `zh-CN`.
- `zh-TW`, `zh-Hant`, `zh-HK`, and `zh-MO` map to `zh-TW`.
- `ja` and Japanese tags map to `ja-JP`.

When no supported locale can be found, the product default is `en-US`.

## Frontend Framework

The web app will use `i18next` and `react-i18next`.

The initial i18n module will live under `apps/web/src/lib/i18n`:

- `locales.ts`: localStorage key, browser language detection, and locale persistence.
- `resources.ts`: static resources for the four supported locales.
- `i18n.ts`: i18next initialization.
- `i18n-provider.tsx`: React provider and app initialization boundary.

The localStorage key will be `rolesta.locale`.

Frontend language selection order:

1. Read `localStorage["rolesta.locale"]`.
2. If no stored value exists, normalize `navigator.languages`.
3. Use `en-US` when no supported locale is found.

The app will expose a small hook or module API for language switching. When the user changes language, the app updates i18next and writes the new locale to localStorage.

## API Request Language

`apps/web/src/lib/api/client.ts` will send the active locale with each request through `Accept-Language`.

The API client currently creates a shared `openApiClient`. To keep request language current after user switching, the header will be produced at request time. An `openapi-fetch` request middleware will read the active i18next language before each request and set:

```http
Accept-Language: zh-CN
```

This keeps the backend independent from frontend storage details.

## Error Envelope

The envelope shape stays simple:

```ts
type ApiSuccessEnvelope<TData> = {
  code: 'SUCCESS';
  msg: string;
  data: TData;
};

type ApiErrorEnvelope<TParams extends Record<string, unknown> = Record<string, unknown>> = {
  code: ErrorCode;
  msg: string;
  data: TParams;
};
```

For error envelopes, `data` contains translation parameters.

Example without parameters:

```json
{
  "code": "VALIDATION_FAILED",
  "msg": "请求参数无效。",
  "data": {}
}
```

Example with parameters:

```json
{
  "code": "PASSWORD_TOO_SHORT",
  "msg": "Password must be at least 8 characters.",
  "data": {
    "min": 8
  }
}
```

One `ErrorCode` maps to one user-visible error template. If two errors need different text, they need distinct codes.

## Shared Error Codes

`packages/shared/src/errors.ts` remains the source of truth for API error codes. It should grow from a generic set into a product-level catalog.

Current codes remain valid:

- `UNAUTHENTICATED`
- `FORBIDDEN`
- `NOT_FOUND`
- `VALIDATION_FAILED`
- `INTERNAL_ERROR`

Business errors should add precise codes such as:

- `INVALID_CREDENTIALS`
- `ADMIN_ALREADY_EXISTS`
- `PASSWORD_TOO_SHORT`

The web app will render API errors with:

```ts
t(`errors.${error.code}`, error.envelope.data)
```

The backend will render `msg` from the same conceptual key:

```ts
errors.${code}
```

Frontend and backend resources can be stored separately, but tests must ensure every `ErrorCode` has translations for all supported locales on both sides.

## Backend Error Handling

`apps/api` will add an API error model that carries:

- HTTP status.
- `ErrorCode`.
- translation params for envelope `data`.

Controllers and services should throw this error model for expected API failures. Generic Nest or unknown exceptions still pass through the global `ApiExceptionFilter`, which converts them into `INTERNAL_ERROR` with empty params.

`ApiExceptionFilter` will:

1. Read `Accept-Language` from the request.
2. Normalize it to a supported locale.
3. Extract `ErrorCode` and params from the thrown API error.
4. Translate `msg` in that locale.
5. Return the existing envelope shape.

`ValidationPipe` will use an exception factory that converts DTO validation failures into:

```json
{
  "code": "VALIDATION_FAILED",
  "msg": "...",
  "data": {}
}
```

Field-level validation can be added later with a structured `data.fields` array. The initial implementation only needs the general validation error, which avoids leaking class-validator's default English text.

## Frontend UI Text

All user-visible UI text in `apps/web` must use i18next resources. Initial migration includes:

- `Login`
- `Create admin`
- `Chat workbench`
- request failure text in form/API helpers

Resource keys should follow a domain-oriented structure:

```ts
{
  common: {
    errors: {
      requestFailed: 'The request failed.'
    }
  },
  auth: {
    login: {
      title: 'Login'
    },
    setup: {
      title: 'Create admin'
    }
  },
  chats: {
    workbench: {
      title: 'Chat workbench'
    }
  },
  errors: {
    VALIDATION_FAILED: 'The request parameters are invalid.'
  }
}
```

The `errors` namespace uses API `ErrorCode` values directly.

## Form Validation

Frontend form validation should avoid library default text. Zod schemas that produce user-facing messages should receive translated text from the call site or use error keys that the UI translates.

For the initial login schema, messages such as invalid email and minimum password length should be driven by i18n resources.

## Data Flow

First visit:

1. Browser exposes `navigator.languages`.
2. Web app normalizes the first supported locale.
3. i18next initializes with that locale.
4. API requests include `Accept-Language`.
5. Backend returns localized `msg`.

User changes language:

1. Web app updates i18next language.
2. Web app writes `rolesta.locale` to localStorage.
3. New API requests include the selected locale.
4. Existing frontend-rendered API errors can re-render from `code` and `data`.

API error display:

1. Backend returns `code`, localized `msg`, and params in `data`.
2. `requestApi` throws `ApiError` with the envelope.
3. UI helpers translate `errors.${code}` with `data`.
4. If a non-API request failure happens before a response, UI uses `common.errors.requestFailed`.

## Testing

Shared package tests:

- Locale normalization maps supported browser/request values correctly.
- Unsupported values use `en-US`.
- Shared `ErrorCode` exports remain the single list used by web and API translation coverage tests.

Frontend tests:

- Initial language uses localStorage when present.
- Initial language uses browser language when localStorage is empty.
- Language switching writes `rolesta.locale`.
- API client sends `Accept-Language`.
- API error helpers translate from `code` and `data`.
- Existing pages render translated titles.

Backend tests:

- `ApiExceptionFilter` returns localized `msg` for each supported locale.
- Missing or unsupported `Accept-Language` uses `en-US`.
- DTO validation returns `VALIDATION_FAILED` and localized `msg`.
- Unknown exceptions return `INTERNAL_ERROR`, localized `msg`, and empty error params.

## Implementation Order

1. Add shared locale and error envelope typing updates.
2. Add frontend i18next setup and resources.
3. Wire frontend locale persistence and API `Accept-Language`.
4. Migrate current frontend text and form errors.
5. Add backend locale parsing, error resources, and API error model.
6. Update `ApiExceptionFilter` and `ValidationPipe`.
7. Add tests across shared, web, and API packages.
8. Run lint, typecheck, and targeted tests.

## Acceptance Criteria

- The web app supports English, Simplified Chinese, Traditional Chinese, and Japanese.
- First visit follows the browser language when it is supported.
- User language selection persists in localStorage.
- API requests include the selected language.
- Backend error responses include localized `msg`.
- Frontend displays API errors from `code` and `data`.
- Current UI text no longer contains hard-coded user-visible strings.
- Tests fail when an API `ErrorCode` lacks a supported locale translation.
