# Chat 管理功能设计

## 目标

本阶段为 Chat 模块建立可用的会话管理能力，覆盖创建、分页列表、筛选、查看、编辑和删除。会话可以关联对话角色、Persona、预设和模型连接。本阶段不包含消息、消息输入、LLM 调用、上下文组装或世界书关联。

## 领域模型

聊天会话归创建它的用户所有，且始终仅对所有者可见。会话包含以下字段：

- `id`
- `ownerUserId`
- `title`
- `chatCharacterId`
- `personaCharacterId`
- `presetId`
- `modelProviderId`
- `createdAtMs`
- `updatedAtMs`

创建会话时必须选择一个当前用户可见的对话角色。角色可以来自用户自己拥有的角色卡，也可以来自其他用户的公开角色卡。创建后允许替换对话角色，用户不能主动清空该关联。

Persona、预设和模型连接均为可选关联。Persona 与模型连接必须归会话所有者所有；预设可以由会话所有者拥有，也可以是其他用户的公开预设。用户可以更换或清空这三个关联。

标题无需唯一，最长 512 个字符。创建时留空会使用当时的对话角色名称。创建成功后标题独立存续，角色改名或更换不会自动修改标题，编辑标题时不允许为空。

## 资产默认值

创建表单读取当前用户的默认 Persona、预设和模型连接，并将它们作为一次性预填值。表单提交后，会话保存自己的独立关联；用户以后修改默认值不会影响已有会话。

选择自有预设时，如果预设已经关联模型连接，表单使用该关联覆盖用户默认模型连接。用户可以在提交前继续更换或清空模型连接。公开预设不暴露原作者的模型连接，选择公开预设时继续保留当前表单中的模型连接。

服务端接收表单最终选中的资产 ID，并重新验证所有访问权限。默认值优先级属于创建表单行为，不进入 Chat 领域模型。

## 生命周期

聊天会话使用硬删除，并在删除前要求用户确认。本阶段不提供归档、置顶、批量操作、软删除、公开或分享能力。

资产删除时，会话继续存在，对应关联被清空：

- 删除角色卡会清空它作为对话角色或 Persona 的关联。
- 删除预设会清空预设关联。
- 删除模型连接会清空模型连接关联。

其他用户的公开角色卡或公开预设转为私有后，Chat 模块清除当前用户会话中对应的越权关联。资产所有者自己的会话关联继续保留。角色关联被系统清除后，会话允许暂时缺少对话角色，用户可以通过编辑会话重新选择角色。

会话只保存资产 ID，不保存角色名称、头像、预设名称或模型连接名称的快照。接口在读取时返回当前仍有效的资产摘要。

## 数据库

新增 `chats` 表。四个资产关联均使用可空外键，删除行为为 `ON DELETE SET NULL`。`owner_user_id` 使用 `ON DELETE CASCADE`。

建议索引覆盖：

- `owner_user_id, updated_at_ms, id`
- `owner_user_id, created_at_ms, id`
- `owner_user_id, title, id`
- `owner_user_id, chat_character_id`

列表排序始终追加 `id` 作为稳定次序。标题关键词使用当前 SQLite 列表查询约定实现。

## 后端模块

`apps/api/src/chats` 按现有业务模块结构补充：

- `domain`：Chat 对象、创建与编辑规则。
- `application`：创建、列表、详情、更新、删除用例，以及资产可见性事件监听器。
- `ports`：`ChatStore` 与 Chat 所需的资产访问能力。
- `persistence`：Kysely Store、行映射和资产访问实现。
- `http`：Controller、请求 DTO、响应 DTO 和错误映射。
- `testing`：用例测试需要的明确 fake 或 builder。

Chat 应用层通过窄 port 验证资产访问权，不导入其他模块的 persistence。模块装配层绑定具体实现。

角色卡和预设模块在资产从公开转为私有时发布过去式领域事件。Chat 监听器在当前 Unit of Work 中清除其他用户会话的关联。监听器失败会使资产更新事务回滚，保持可见性与引用一致。

## HTTP API

新增以下端点：

- `POST /chats`
- `GET /chats`
- `GET /chats/:id`
- `PATCH /chats/:id`
- `DELETE /chats/:id`

创建请求包含必填的 `chatCharacterId`，以及可选的 `title`、`personaCharacterId`、`presetId` 和 `modelProviderId`。更新请求支持标题与四项资产关联；`chatCharacterId` 只接受新的角色 ID，另外三个关联接受 `null` 以清空。

列表接口支持：

- 标题关键词。
- 单一角色筛选器对应的“全部”“缺少角色”“指定角色”三种状态。
- `createdAt`、`updatedAt`、`title` 排序。
- 升序或降序。
- `pageIndex` 和 `pageSize`。

列表默认按 `updatedAt` 倒序，每页 20 条。Web 允许选择 10、20、50 或 100 条。

列表响应提供会话 ID、标题、对话角色摘要和更新时间。详情响应增加 Persona、预设、模型连接摘要及创建时间。响应不包含角色卡正文、预设正文、模型凭据或其他私有资产内容。

查询不存在或不属于当前用户的 Chat 时返回相同的 404。创建或更新引用无权访问、已经删除或已经失效的资产时，返回稳定的 Chat application error，由 HTTP 层映射为对应错误响应。

接口完成后重新生成 `apps/api/openapi.json` 与 `apps/web/src/lib/api/generated/schema.ts`，Web 只使用生成契约。

## Web 交互

现有左侧 Chat Context 面板改为两个 Tab：

- “聊天列表”：创建、搜索、筛选、排序、分页和会话选择。
- “当前聊天”：查看当前会话完整配置并执行编辑、删除。

访问 `/app` 时默认打开“聊天列表”。用户选择会话或创建成功后进入 `/app/chats/:chatId` 并切到“当前聊天”。用户可以随时切回列表。窄屏选择会话后关闭左侧抽屉，使中央区域重新可见。

聊天列表适配左栏 224–448px 的宽度范围。每条记录展示标题、对话角色头像与名称、最近更新时间，以及缺少角色状态。当前会话高亮，条目菜单提供编辑和删除。Persona、预设和模型连接在“当前聊天”Tab 展示。

筛选区包含标题搜索、单一对话角色筛选器和排序菜单。筛选或排序改变时回到第一页。分页复用 `features/assets/components/page-controls.tsx`，保持其他资产管理器的页码和每页数量交互。

创建与编辑共用一个对话框。桌面端居中展示，移动端占用大部分可用空间。资产选择器支持搜索；对话角色和预设搜索本人及公开资产，Persona 与模型连接只搜索本人资产。提交成功后关闭对话框，刷新列表与详情，并进入对应会话。

“当前聊天”Tab 展示标题、ID、四项资产摘要、创建时间和更新时间，并提供编辑与删除命令。没有活动会话时显示“未选择聊天”。

中央区域在存在活动会话时只显示会话名称与 ID。组件源码添加 `FIXME`，标记未来消息界面的替换位置。没有活动会话时保留现有工作区首页。

## 错误状态

列表加载失败时，左栏显示错误状态与重试命令。活动会话详情返回 404 时，Web 跳转到 `/app` 并切回聊天列表。删除当前会话后执行同样的导航。

创建或编辑期间发生资产失效或访问权变化时，对话框保留用户已经填写的内容，并在对应字段显示错误。提交期间禁用重复提交。

## 测试与验证

后端测试覆盖：

- 标题默认值、长度与编辑规则。
- 创建、读取、更新和删除的所有权边界。
- 对话角色、Persona、预设和模型连接的访问权校验。
- 角色筛选、关键词、排序与分页组合。
- 资产删除后的外键清理。
- 角色卡和预设可见性变化后的事务内关联清理。
- Controller 响应与稳定错误映射。

Web 测试覆盖：

- Chat API 生成类型的使用。
- 列表查询、分页与缓存失效。
- 左栏 Tab 自动切换。
- 创建、编辑和删除交互。
- 默认资产与预设模型连接优先级。
- 活动会话 404 和删除后的导航。
- 桌面与窄屏左栏布局。

验证执行受影响 workspace 的测试、类型检查和 lint，并检查 OpenAPI 生成文件。最后通过浏览器验证桌面与移动端的创建、筛选、分页、查看、编辑和删除流程。
