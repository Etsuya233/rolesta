# Rolesta Project Skeleton Design

Date: 2026-06-30

## Purpose

This spec defines the first project skeleton for Rolesta. The goal is to make later feature work smooth, readable, and maintainable while keeping personal deployment lightweight.

Rolesta will start as a frontend/backend separated TypeScript monorepo. The first implementation phase should establish project structure, development scripts, database wiring, API conventions, frontend conventions, and test foundations. It should not attempt to implement the full product surface.

## Chosen Stack

- Monorepo: pnpm workspaces and Turborepo.
- Backend: NestJS, TypeScript, REST APIs, OpenAPI contract generation.
- Frontend: React, Vite, TypeScript.
- UI: Tailwind CSS, shadcn/ui, Radix.
- Database access: Kysely.
- Default database: SQLite.
- Future database targets: PostgreSQL and MySQL through small dialect adjustments and selective repository overrides.
- Authentication: local accounts with Session Cookie.
- Streaming chat transport: Server-Sent Events.
- Server state in the frontend: TanStack Query.
- Local UI state: React state, React Context, and limited Zustand where shared client state is genuinely useful.
- Delivery: local Node.js runtime with SQLite by default; Docker Compose as an optional entry point.

## Repository Layout

The monorepo should use this top-level shape:

```text
apps/
  api/
  web/
packages/
  config/
  db/
  shared/
docs/
  superpowers/
    specs/
```

### apps/api

`apps/api` contains the NestJS backend. It owns HTTP controllers, authentication, user isolation, business services, repositories, model provider integration, SSE streaming, import/export orchestration, and generation debug persistence.

### apps/web

`apps/web` contains the React workbench. It owns routing, layout, UI components, feature modules, form handling, API client integration, SSE client integration, and frontend state composition.

### packages/config

`packages/config` contains shared TypeScript, ESLint, Prettier, Vitest, and related tooling configuration.

### packages/db

`packages/db` contains Kysely table types, migrations, dialect creation, and database test utilities. It should stay infrastructure-focused and avoid business repository logic.

### packages/shared

`packages/shared` contains stable types, enums, constants, error codes, pagination helpers, compatibility-format types, variable directory definitions, and pure utility functions shared by frontend and backend.

Generated frontend API types should come from OpenAPI. The project should avoid hand-maintaining the same DTOs in backend, shared, and frontend code.

## Backend Architecture

The backend is organized by business modules:

- `auth`
- `users`
- `characters`
- `worldbooks`
- `presets`
- `model-profiles`
- `chats`
- `generation-debug`
- `health`

Each module should normally contain:

- `*.controller.ts` for HTTP routing, status codes, request parsing, and OpenAPI annotations.
- `*.service.ts` for business rules and cross-repository orchestration.
- `*.repository.ts` for Kysely data access.
- `dto/` for request and response DTOs.
- `types/` or `domain/` for module-local types and pure domain helpers.

Controllers must not access the database directly. Repositories must not make HTTP decisions. Services coordinate business behavior and enforce module rules.

Cross-module calls should normally go through service APIs. If a module needs data owned by another module, the dependency should be explicit in the consuming service.

The chat generation path should have a dedicated orchestration service. It should coordinate message persistence, role selection, preset assembly, worldbook matching, variable replacement, history truncation, model invocation, SSE events, assistant message persistence, and debug snapshot creation.

## Database Design

SQLite is the only database that the first skeleton must run against. PostgreSQL and MySQL compatibility are design constraints, not first-phase runtime requirements.

Business repositories should default to one canonical Kysely implementation. That implementation should use portable SQL patterns and avoid deep database-specific behavior.

Repository methods should expose stable business operations, such as:

- `findByOwnerId`
- `findByIdForOwner`
- `searchCharacters`
- `appendMessage`
- `listMessagesPage`

If a future database needs a dialect-specific query for correctness or performance, the project may add a focused override such as `PostgresCharacterRepository` or `SqliteChatRepository`. Overrides should replace only the methods that need special handling.

NestJS providers should select the concrete repository implementation based on the configured database dialect.

### SQLite Constraints

SQLite should remain simple in the first phase:

- IDs are generated by the application as strings, preferably UUID or ULID.
- Timestamps use a consistent application-level representation, preferably ISO strings.
- Booleans are represented consistently through the Kysely type layer.
- JSON compatibility snapshots are stored as text and parsed/stringified in application code.
- Fields that need filtering or sorting are normal columns.
- Complex JSON path queries, database triggers, stored procedures, full-text indexes, and advanced database-specific types are out of scope for the first skeleton.

All user-owned asset tables should include `owner_user_id`. Repository methods that read or mutate user-owned data should require the current user identity or owner id and apply it as a mandatory condition.

Sensitive model credentials must never be returned to the frontend. The schema should distinguish encrypted secret storage from a safe "secret is configured" indicator.

## API Contract

The backend exposes REST APIs and generates an OpenAPI document from NestJS controller and DTO metadata.

The frontend consumes that OpenAPI document with `openapi-typescript`. The generated output provides TypeScript types. A lightweight fetch wrapper in `apps/web` should use these types to keep request and response handling explicit.

The frontend should not assemble raw endpoint URLs in feature components. Feature code should call typed API helpers and React hooks.

This keeps the contract observable:

1. Backend DTOs and controllers describe the API.
2. OpenAPI JSON is generated from the backend.
3. Frontend types are generated from OpenAPI.
4. TypeScript catches frontend usage drift after API changes.

## Authentication and User Isolation

Authentication uses local accounts and Session Cookie.

The first product flow includes:

- First administrator setup.
- Login.
- Logout.
- Current user lookup.
- Session-backed API access.

All business requests resolve the current user from the session. User-owned data access must be scoped by owner id at the service and repository boundary.

CSRF policy should be explicit in the backend configuration. The first skeleton should leave a clear middleware/configuration location for non-GET request protection.

## Frontend Architecture

The frontend starts as a workbench application, not a marketing page. The main authenticated route should lead users into the Rolesta workspace.

Recommended structure:

```text
src/
  app/
  components/
  features/
    auth/
    characters/
    worldbooks/
    presets/
    model-profiles/
    chats/
    debug/
  lib/
    api/
    forms/
    state/
  styles/
```

### Routing

Initial routes:

- `/setup` for first administrator setup.
- `/login` for login.
- `/app` for the main workbench.
- `/app/chats/:chatId` for deep linking to a chat while keeping the workbench layout.
- `/app/assets/*` for direct links into asset management areas.
- `/app/settings/*` for model profiles, account settings, and theme settings.

### Workbench Layout

Desktop layout should be compact and multi-column:

- Left area: sessions and asset navigation.
- Center area: message list and composer.
- Right area: selected character, worldbooks, preset, model profile, runtime information, and debug panels.

Narrow screens should prioritize the message list and composer. Navigation, assets, runtime information, and debug panels should move into drawers or similar secondary surfaces.

### State Management

TanStack Query manages server state: current user, character lists, worldbooks, presets, model profiles, chat messages, and debug records.

Local UI state uses React state or Context first. Zustand may be used for small shared UI state such as active workspace selection, drawer state, or command palette state if prop drilling becomes noisy.

React Hook Form and Zod should handle form state and client-side validation.

SSE handling should be isolated in a client module that parses events, appends streamed text, tracks failure/completion status, and coordinates cache updates with TanStack Query.

## Import, Compatibility, and Debugging

SillyTavern-compatible imports should be handled by explicit parser modules. Parsers convert external formats into Rolesta internal objects and preserve raw import snapshots.

The first skeleton should reserve places for:

- Character Card JSON import.
- PNG character card import.
- Worldbook import.
- Preset import.
- Compatibility export.
- Generation debug snapshots.

Parser logic should be pure and testable. External data validation should happen at parser boundaries.

Generation debug data should record enough structured information to explain a generated response:

- Request message summary.
- Matched worldbook entries.
- Variable replacements.
- Context fragments.
- Estimated context length.
- History truncation details.
- Failure reason when generation fails.

## Testing Strategy

Testing should scale with risk and keep the skeleton practical:

- `packages/shared`: unit tests for pure helpers, compatibility parsers, variable replacement, pagination helpers, and error helpers.
- `packages/db`: migration tests, temporary SQLite database tests, and repository integration tests.
- `apps/api`: service unit tests, repository integration tests, and focused REST API e2e tests.
- `apps/web`: focused component tests for important forms and state transitions, plus Playwright tests for core flows.
- Chat streaming: backend tests with a mock model provider to verify SSE event order; frontend tests for token assembly, failure handling, and final cache synchronization.

The first CI should install dependencies, run type checking, linting, tests, and builds. Database tests should use temporary SQLite and require no external services.

## Scripts

The root workspace should expose at least:

- `pnpm dev`
- `pnpm build`
- `pnpm test`
- `pnpm test:e2e`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm db:migrate`
- `pnpm db:seed`
- `pnpm openapi:generate`

`pnpm dev` should start the API and web app together through Turborepo.

## Delivery

The default local development and deployment path is Node.js plus a local SQLite database file.

The repository should include `.env.example` covering:

- SQLite database path.
- Session secret.
- API host and port.
- Frontend API base URL.
- Model provider development settings.

Docker Compose is optional. It may include API and web services and mount a SQLite data directory. PostgreSQL and MySQL services should not be default dependencies for the first skeleton.

## First Skeleton Scope

The first skeleton should establish:

- Monorepo workspace.
- Shared tooling packages.
- NestJS API app.
- React Vite web app.
- Kysely SQLite connection and migration path.
- OpenAPI generation path.
- Frontend typed API generation path.
- Session auth scaffolding.
- Module directory conventions.
- Basic health endpoint.
- Basic test and CI script structure.

The first skeleton should leave feature implementations thin. Full character management, worldbook behavior, preset editing, model calls, chat generation, and debug views belong to later feature phases.

## Decisions

- Use TypeScript across frontend and backend.
- Keep backend and frontend as separate apps inside one monorepo.
- Use NestJS for backend module conventions.
- Use React and Vite for frontend workbench development.
- Use Kysely with SQLite first.
- Keep repository code portable by default and add dialect overrides only when needed.
- Use REST and OpenAPI as the frontend/backend contract.
- Use `openapi-typescript` for generated frontend types and a small fetch wrapper for API calls.
- Use SSE for first streaming chat generation.
- Use local accounts and Session Cookie.
- Use Tailwind CSS, shadcn/ui, and Radix for UI foundations.
- Use TanStack Query for server state.
- Keep Docker Compose optional.
