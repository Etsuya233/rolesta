# 文件服务设计

## 背景与目标

Rolesta 需要统一保存和读取文件，第一批调用方是用户头像与角色头像。文件服务需要在部署级别支持本地文件系统和数据库两种存储后端，并为未来接入 S3 或 MinIO 保留清晰的适配器边界。

本次目标：

- 建立通用文件资源模型与二进制存储端口。
- 通过部署配置选择本地存储或数据库存储。
- 完整接入用户头像和角色头像的上传、裁剪、替换、读取与删除。
- 为头像生成 64x64、128x128、256x256 三档 WebP 派生图。
- 为前端提供稳定、可长期缓存的公开资源 URL 与结构化 `srcset` 数据。
- 通过延迟清理处理未完成上传和失去引用的文件。

本次不包含：

- 面向前端开放任意文件的通用上传 API。
- S3 或 MinIO 适配器。
- 跨存储后端自动迁移文件。
- 动画头像、动态图片变换或内容寻址去重。
- 多实例清理任务协调。

## 核心决策

- 文件同时支持用户私有和公开可见性。
- 存储后端按部署实例统一选择，每个文件不单独选择后端。
- 文件对象不可变；替换内容会创建新对象并更新业务引用。
- 文件元数据始终保存在数据库，二进制内容由存储端口管理。
- 原始头像保持私有，三档派生头像公开读取。
- 浏览器本地预览和选择裁剪区域，服务端完成可信的裁剪、缩放与编码。
- 失去引用的资源延迟 24 小时清理。

## 架构

新增独立的 `FilesModule`。该模块只负责文件生命周期、图片处理和存储介质隔离，不理解用户头像或角色头像的业务规则。

### 文件模块组件

- `FileMetadataStore`：保存逻辑资源、文件对象、所有权、可见性、派生关系与生命周期。
- `FileContentStore`：保存、读取和删除二进制内容。
- `LocalFileContentStore`：把内容写入配置的本地目录。
- `DatabaseFileContentStore`：把内容写入数据库内容表。
- `ImageProcessor`：提供格式识别、尺寸读取、裁剪、缩放和 WebP 编码等通用图片能力。
- `FileReadService`：执行读取授权，返回流、媒体类型、长度和缓存元数据。
- `OrphanFileCleanupService`：清理超过保留期的未完成资源和孤立资源。

未来的 `S3FileContentStore` 实现相同端口。调用方及 HTTP 读取层无需感知实际存储介质。

### 业务模块职责

用户头像用例属于 `UsersModule`，角色头像用例属于 `CharactersModule`。它们负责：

- 验证当前用户与目标业务对象的权限。
- 定义头像文件格式、大小、尺寸、裁剪和派生规格。
- 调用通用 `ImageProcessor` 生成派生内容。
- 调用文件服务保存资源和对象。
- 在业务事务中切换头像引用并孤立旧资源。

`FilesModule` 不反向依赖用户或角色模块，避免模块循环依赖。后续附件或其他图片可以复用文件端口，并定义自己的业务约束。

## 数据模型

### `file_resources`

一个记录表示一个逻辑文件资源。

| 字段             | 含义                                                       |
| ---------------- | ---------------------------------------------------------- |
| `id`             | 资源 ID                                                    |
| `owner_user_id`  | 所有者用户 ID                                              |
| `purpose`        | 调用方提供的用途标识，如 `user-avatar`、`character-avatar` |
| `status`         | `pending`、`active` 或 `orphaned`                          |
| `orphaned_at_ms` | 进入孤立状态的时间，其他状态为空                           |
| `created_at_ms`  | 创建时间                                                   |

### `file_objects`

一个记录表示逻辑资源下的一个不可变二进制对象。

| 字段                 | 含义                                                    |
| -------------------- | ------------------------------------------------------- |
| `id`                 | 公开 URL 使用的文件 ID                                  |
| `resource_id`        | 所属逻辑资源                                            |
| `role`               | 调用方定义的对象角色，如 `original`、`64`、`128`、`256` |
| `visibility`         | `private` 或 `public`                                   |
| `media_type`         | 服务端识别的媒体类型                                    |
| `byte_size`          | 内容长度                                                |
| `width`、`height`    | 图片尺寸，非图片允许为空                                |
| `content_hash`       | 内容哈希，用于 ETag 与完整性检查                        |
| `storage_key`        | 存储适配器使用的不透明键                                |
| `original_file_name` | 可选的原始文件名，仅作为元数据                          |
| `created_at_ms`      | 创建时间                                                |

`storage_key` 和文件 ID 均由服务端生成。用户提供的文件名不参与路径构造。

### `file_contents`

该表供数据库内容适配器使用。

| 字段          | 含义                   |
| ------------- | ---------------------- |
| `storage_key` | 与文件对象对应的存储键 |
| `content`     | 二进制内容             |

### 业务引用

- `users.avatar_resource_id` 可为空，外键指向 `file_resources.id`。
- `characters.avatar_resource_id` 可为空，外键指向 `file_resources.id`。

一个头像资源包含一个私有原图对象，以及 64、128、256 三个公开 WebP 对象。

## 配置与存储布局

```env
FILE_STORAGE_DRIVER=local
FILE_STORAGE_LOCAL_DIRECTORY=.data/files
FILE_ORPHAN_RETENTION_HOURS=24
```

`FILE_STORAGE_DRIVER` 只接受 `local` 或 `database`。本地驱动按存储键前缀分层保存文件，避免单个目录积累过多条目。

运行中的服务只访问当前配置的驱动，也不在每个文件对象中记录驱动类型。切换驱动前必须通过独立的运维迁移流程复制和校验全部内容；本次不实现该迁移工具。

## 头像上传与关联流程

业务端点：

- `PUT /users/me/avatar`
- `DELETE /users/me/avatar`
- `PUT /characters/{characterId}/avatar`
- `DELETE /characters/{characterId}/avatar`

上传使用 `multipart/form-data`，包含原图以及归一化裁剪区域 `x`、`y`、`width`、`height`。

写入流程：

1. 业务用例验证身份、对象权限、文件大小和请求参数。
2. `ImageProcessor` 根据文件签名识别格式、解码图片并验证尺寸。
3. 业务用例要求 `ImageProcessor` 根据裁剪区域生成 64x64、128x128、256x256 三档 WebP。
4. 文件服务创建状态为 `pending` 的资源和四个文件对象元数据。
5. 当前内容存储适配器写入原图和三个派生对象。
6. 在一个数据库事务中将新资源设为 `active`、更新业务对象的头像引用，并把旧资源设为 `orphaned`。
7. API 返回新头像资源及三档公开 URL。

处理或写入失败时，已有头像引用保持不变。未完成的 `pending` 资源由清理任务处理。

删除头像时，业务事务清空头像引用并将旧资源设为 `orphaned`。请求流程不立即删除二进制。

## 文件读取

统一读取端点：

```text
GET /files/{fileId}/content
```

读取规则：

- `public` 对象允许匿名读取。
- `private` 对象要求登录，并且当前用户必须是资源所有者。
- 只有 `active` 资源下的对象可读取。
- 不存在、不可读取、`pending` 或 `orphaned` 统一返回 `404`，避免泄露资源是否存在。
- 本地适配器提供文件流；数据库适配器将二进制包装为可读流。

公开不可变对象返回：

- `Cache-Control: public, max-age=31536000, immutable`
- `ETag`
- `Content-Type`
- `Content-Length`
- `X-Content-Type-Options: nosniff`

端点支持 `If-None-Match`，哈希匹配时返回 `304`。

## API 响应与前端获取

用户和角色 DTO 使用相同的头像结构：

```ts
type AvatarResource = {
  resourceId: string;
  sources: {
    64: string;
    128: string;
    256: string;
  };
};
```

服务端负责生成完整的对象 URL。前端不拼接文件路径，也不感知存储后端。

展示组件基于现有 Radix Avatar，并通过 `src`、`srcSet` 和 `sizes` 交给浏览器选择资源。列表优先使用 64 档，常规编辑界面使用 128 档，高密度或较大预览可选择 256 档。头像更新后文件 ID 随内容变化，不添加缓存破坏参数。

## 前端头像编辑

用户资料和角色编辑器复用同一头像编辑流程：

1. 用户选择 JPEG、PNG 或 WebP。
2. 浏览器通过本地 Object URL 打开固定 1:1 的裁剪对话框。
3. 用户拖动和缩放图片，通过圆形遮罩预览头像结果。
4. 确认后提交原文件和归一化裁剪坐标；前端不生成派生图片。
5. 上传期间禁止重复提交。
6. 成功后用响应数据更新 React Query 缓存并立即显示新 URL。
7. 取消或关闭时释放 Object URL，不产生服务端资源。

删除头像需要明确确认，成功后显示现有文字首字母占位。上传校验错误显示在裁剪对话框中，基础设施错误使用全局通知。

## 校验与错误处理

上传校验：

- 文件最大 10 MB，超限返回 `413`。
- 只接受文件签名为 JPEG、PNG、WebP 的静态图片，不信任扩展名或客户端 MIME；不支持的格式返回 `415`。
- 图片解码失败返回 `422`。
- 任一边不得超过 8192 px，总像素不得超过 40 MP，超限返回 `422`。
- 裁剪坐标必须位于 `[0, 1]`，宽高大于零，且裁剪区域完整位于图片内；无效参数返回 `400`。
- 派生 WebP 移除 EXIF 等图片元数据。

存储后端不可用映射为 `503`，图片处理错误映射为明确的 `422` 应用错误。错误类型沿用仓库现有 `RolestaError` 和 HTTP error mapper 结构。

## 孤立资源清理

`OrphanFileCleanupService` 在服务启动时执行一次，随后每小时执行：

1. 批量查找超过 24 小时的 `pending` 和 `orphaned` 资源。
2. 删除资源下的全部内容对象。
3. 全部内容删除成功后删除对象与资源元数据。
4. 删除失败的资源保留元数据，下一轮继续处理。

第一版按单实例部署设计，不增加分布式锁。多实例部署需要在后续引入任务租约或独立 worker。

## 测试策略

### 单元测试

- `ImageProcessor`：格式识别、尺寸限制、裁剪边界、三档尺寸、WebP 输出和元数据移除。
- 头像用例：权限、资源创建、引用切换、旧资源孤立化，以及失败时保留已有头像。
- `FileReadService`：公开读取、私有所有者读取、越权隐藏、资源状态和 ETag 条件请求。
- 清理服务：24 小时边界、两类待清理状态，以及删除失败后的后续重试。

### 存储契约测试

同一套契约测试分别运行 `LocalFileContentStore` 和 `DatabaseFileContentStore`，覆盖保存、流式读取、字节一致性、删除和重复键冲突。本地测试使用临时目录，数据库测试使用仓库现有测试数据库工具。

### API 集成测试

- 用户头像与角色头像的上传、替换和删除。
- 非角色所有者无法修改头像。
- 三种输入格式都生成正确的三档 WebP。
- 文件大小、边长、总像素、文件签名和裁剪参数限制。
- 公开派生头像允许匿名读取，私有原图只允许所有者读取。
- 缓存、类型、长度和安全响应头。

### 前端测试

- 选择文件后打开裁剪对话框。
- 取消不上传，确认提交原文件与归一化坐标。
- 上传成功后更新头像及 `srcset`。
- 错误展示、删除流程和占位内容。
- 用户与角色入口复用相同交互。

## 验收标准

- 本地和数据库两种驱动通过相同的存储契约测试。
- 两种驱动下均能完成用户头像和角色头像的上传、匿名派生图读取、替换与延迟删除。
- 公开头像具备稳定 URL、长期不可变缓存和条件请求能力。
- 前端列表与编辑页使用对应尺寸的资源，无需额外查询文件元数据。
- OpenAPI 文档和生成的 Web 类型与新增端点、DTO 保持一致。
- API 与 Web 类型检查、相关测试和构建全部通过。
